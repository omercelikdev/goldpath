# RFC: Goldpath Migrations & Upgrade Story (H1)

> The CRITICAL hardening item (ledger, honest assessment 2026-07-08): package-contributed
> tables (qrtz_, archive, bulk, notification, campaign, audit...) need a proven path for
> (a) enabling a feature on a LIVE system and (b) upgrading BETWEEN Goldpath versions when a
> package changes its model. Today the template SAYS "production applies the CI migration
> bundle" — and nothing anywhere produces one. Status: v0.1 DRAFT for approval (2026-07-10).

## 1. Scope / Non-Goals

**In scope:**
- The schema lifecycle of a generated app across its WHOLE life: first deploy, feature
  enablement on a live system (`goldpath add feature` → new package tables), Goldpath package
  upgrades that change a package's model contribution, multi-head solutions (api +
  workers sharing one database).
- The `goldpath db` verb family (the CLI wraps `dotnet ef` the way it wraps everything else:
  thin, deterministic, teaching).
- The migration BUNDLE as the production apply mechanism (self-contained, CI-built).

**Non-goals (deferred with written triggers):**
- Automatic schema-affecting-package detection (model-hash drift between package
  versions) — trigger: the first upgrade incident a CHANGELOG marker would not have
  prevented. V1 is a disciplined CHANGELOG marker + upgrade-guide line (rides H7).
- Zero-downtime/expand-contract choreography — trigger: the first adopter with a
  no-maintenance-window requirement. V1 documents the constraint honestly.
- Data MIGRATIONS (backfills) — the app's domain work; the runbook shows where they hang.
- Oracle — rides the existing provider trigger.

## 2. Decisions (D1–D7)

- **D1 — Migrations live in the APP, never in packages.** A Goldpath package contributes
  MODEL (via `modelBuilder.AddGoldpathX()`); the EF migration is always generated inside the
  app against the app's provider and the app's composed model. Packages shipping
  pre-baked migrations cannot compose (provider differences, feature combinations,
  table-prefix collisions). Consequence: enabling a feature or upgrading a package is
  always followed by ONE app-side migration step — and the tooling makes that step
  impossible to forget (D5).

- **D2 — Development moves from EnsureCreated to `MigrateAsync()`; the template ships
  migration-ready.** EnsureCreated writes no history row: a database born from it can
  never take a migration later — the current template bakes the exact trap H1 exists to
  remove. The template gains the `Microsoft.EntityFrameworkCore.Design` reference and a
  committed INITIAL migration is generated by `goldpath new` itself as a post-generation step
  (the template cannot carry one file per feature combination; the CLI generates it
  against the just-composed model). Development runs `MigrateAsync()` under the same
  GP0302 guard; real environments apply the bundle. GM validates the from-zero migrate
  path — EnsureCreated leaves the golden path entirely.

- **D3 — One table set, ONE migration owner.** In a multi-head solution the Api's
  context OWNS the shared package tables (qrtz_, campaign, ...). A worker context that
  maps the same tables (a jobs-trigger worker maps AddGoldpathJobs to run its own fleet)
  EXCLUDES them from its migrations: `modelBuilder.AddGoldpathJobs(excludeFromMigrations:
  true)` (EF's `SetIsTableExcludedFromMigrations` under the seam — the model is complete,
  the DDL responsibility is not duplicated). The worker still owns migrations for its
  PRIVATE tables (DailyReports). `goldpath add worker --trigger jobs` generates the exclusion
  by default; GP1801 makes the double-ownership mistake visible.

- **D4 — The production apply mechanism is the EF BUNDLE.** CI builds
  `dotnet ef migrations bundle --self-contained` per migration-owning project and ships
  it as an artifact; deployment runs the bundle BEFORE the new app version starts
  (init-container/pre-deploy hook — the runbook shows both). No runtime `Migrate()` in
  production: the app process never holds DDL rights (the security floor the finance
  card assumes).

- **D5 — The `goldpath db` verb family** (wraps `dotnet ef`, multi-context aware):
  - `goldpath db init` — the post-`goldpath new` step (D2): generates the Initial migration for
    every migration-owning project.
  - `goldpath db add <name>` — after `goldpath add feature`/a package upgrade/domain model work;
    knows which projects own migrations and runs against each owner.
  - `goldpath db status` — "the model changed but no migration captures it" (wraps
    `dotnet ef migrations has-pending-model-changes`); `goldpath check` runs it too, so the
    engine round-trip catches a forgotten migration the same way it catches drift.
  - `goldpath db bundle` — what CI runs; also works locally for air-gapped delivery.
  - `goldpath add feature X` ends its NextSteps with the exact `goldpath db add` line — and
    `goldpath check` goes red until it happened.

- **D6 — Package upgrades are CHANGELOG-marked.** Any Goldpath release whose package changes
  a model contribution carries a `[schema]` marker on the CHANGELOG entry and an
  upgrade-guide line ("after updating, run `goldpath db add goldpath-<package>-<version>`").
  V1 discipline + H7's written contract; automated detection stays deferred (§1).

- **D7 — Proof obligations (the item is closed by EVIDENCE, not by tooling existing):**
  1. From-zero: a generated app migrates from an empty database and the smoke suite
     passes (GM shape — replaces the EnsureCreated path).
  2. LIVE feature enablement: app v1 deployed with DATA → `goldpath add feature campaign` →
     `goldpath db add` → bundle applied → old data intact, new tables live, smoke green.
  3. LIVE Goldpath upgrade rehearsal: simulate a package model change → `goldpath db add` → bundle
     applies on a populated database, zero data loss.
  4. Multi-head: api + jobs-worker both migrated; the worker's bundle never touches the
     shared tables (D3 exclusion proven, not assumed).

## 3. Manifest Surface

None in v1 — migrations are not a feature toggle; they are how every feature reaches a
real database. (A future `db.migrations: off` escape hatch is explicitly REJECTED: an app
that opts out of the schema story is not on the golden path.)

## 4. API Surface

- `Goldpath.Data`: `AddGoldpathJobs`/model-contribution extensions gain the
  `excludeFromMigrations` parameter (default `false`) — one mechanical overload per
  module model extension, implemented once in a shared helper.
- `Goldpath.Cli`: the `goldpath db` verb family (D5). No new packages.

## 5. Analyzer Rules

- **GP1801 (warning):** a second DbContext in the solution maps a Goldpath package model
  without `excludeFromMigrations: true` while another context already maps it — double
  DDL ownership of shared tables.
- **GP0302 stays** (runtime schema mutation outside Development) and now also matches
  `MigrateAsync` outside the guard — production applies bundles, processes do not.

## 6. Ops Package

- `docs/ops/migrations-runbook.md`: first deploy, feature-on-live, Goldpath upgrade, rollback
  stance (bundles are forward-only; rollback = restore + roll forward — said honestly),
  init-container and pre-deploy-hook examples, air-gapped delivery via `goldpath db bundle`.
- The CI pipeline gains the bundle-build step next to `pack` (artifact per owner project).

## 7. Test Plan

- CLI unit: `goldpath db` verb goldens (owner discovery, multi-context fan-out, status wiring
  into `goldpath check`, the NextSteps line from `goldpath add feature`).
- Integration (real pg + sqlserver): the four D7 proofs as tests; the live-enablement
  proof reuses the GM harness (generate → deploy v1 → add feature → migrate → smoke).
- Analyzer tests: GP1801 both ways.
